매입거래 부정·이상 탐지 엔진

ERP가 통과시킨 매입거래까지 다중 신호로 재검증해 부정·이상을 잡아내는 외부 감사 분석 도구입니다.


프로젝트 소개

대기업은 SAP·GRC 모듈로 정교한 부정 탐지·감사 통제를 누리지만, 영림원·더존·이카운트 등을 쓰는 중소·중견 기업은 기본 송장검증은 있어도 부정 패턴 탐지(중복·라운드 금액)나 외부 공공 API 거래처 검증 같은 상위 감사 레이어는 누리기 어렵습니다.

이 프로젝트는 SAP 기능을 파이썬으로 복제하는 것이 아닙니다. SAP Clean Core 원칙에 따른 Side-by-Side 외부 확장 패턴 — ERP에서 추출한 거래 데이터에 외부 도구로 다중 신호 분석 레이어를 얹는 구조입니다. 따라서 ERP 종류와 무관하게, ERP가 이미 통과시킨 거래를 더 엄격한 허용오차와 ERP가 교차검증하지 못하는 부정 신호로 독립적으로 재검증합니다.


핵심 메시지: ERP 로직을 복제하는 것이 아니라, ERP가 통과시킨 결과를 외부에서 다시 한 번 검증하는 감사 레이어입니다.




주요 기능 (3축 병렬 검증)

① 거래 정합성 검증 (3-Way Match)

주문서(PO)·입고증(GR)·청구서(Invoice)의 수량·단가·금액을 자동 대조합니다.


단가 검사: 발주 단가 대비 청구 단가가 허용오차(기본 3%)를 벗어나면 적발
수량 검사: 청구 수량이 입고 수량을 초과하면 적발 (수량은 허용오차 없음)
입고기록 검사: 입고 없이 청구만 도착한 건 적발
청구 1건이 PO 여러 건을 묶는 1:N 매칭, 부분입고·분할청구 지원


허용오차는 Strategy 패턴(PercentTolerance / AbsoluteTolerance)으로 분리되어, 코어 코드를 건드리지 않고 검사 강도를 교체할 수 있습니다.

② 부정 패턴 탐지

3-Way Match만으로는 잡히지 않는 부정 신호를 별도로 탐지합니다.


중복 청구 탐지: 같은 거래처가 근접한 일자(기본 7일 이내)에 같은 금액을 두 번 청구하면 적발
라운드 금액 탐지: 결재 한도 회피가 의심되는 금액(예: 정확히 ₩5,000,000)과 일치하면 적발


각 탐지기는 추상 클래스 AnomalyDetector를 상속해 동일한 detect() 인터페이스를 따르므로, 새로운 부정 신호를 손쉽게 추가할 수 있습니다.

③ 거래처 실재성 검증

국세청 사업자등록 상태조회 API를 실시간 호출해 폐업·휴업 거래처를 적발합니다. ERP의 3-Way Match는 통과하지만 거래처가 이미 폐업한 경우 등, ERP가 보지 못하는 사각지대를 차단합니다.

VendorVerifier 추상 인터페이스 + 다중 구현 구조로 설계되어, API 호출이 실패(키 미승인·한도초과·타임아웃)하면 폴백 처리(검증불가)로 코어가 죽지 않게 합니다.


판정 흐름

1. 입력      거래처 마스터(공정위 실데이터) + 거래 3종(합성)
       │
2. 전처리    데이터 로딩 · PO 기준 집계 (1:N · 부분입고 · 분할청구)
       │
3. 검증·탐지 [정합성] [부정 패턴] [거래처 실재성: 국세청 API]  ← 3축
       │
4. 판정      통합 판정 · 사유 자동 분류 (하나라도 적발 → 보류)
       │
5. 출력      엑셀 리포트 (판정결과 · 요약 통계 + 원본 데이터)

검사 결과 하나라도 적발(passed=False)되면 해당 청구는 보류로 판정됩니다.

API 키가 없어도 정합성·부정 패턴 검증은 정상 동작합니다. 거래처 검증만 '검증불가'로 처리됩니다.




거래처 검증 API 설정

국세청 API를 호출하려면 인증키가 필요합니다. 키는 소스에 하드코딩하지 않고 환경변수(.env)로 분리합니다.


공공데이터포털에서 국세청 사업자등록 상태조회 API 활용신청 후 인증키 발급
프로젝트 루트에 .env 파일 생성:


NTS_SERVICE_KEY=발급받은_Decoding_키


.env는 .gitignore에 등록해 저장소에 올라가지 않게 합니다.



API 키가 없거나 호출이 실패하면 해당 거래처는 검증불가로 처리되어, 정합성·부정 패턴 검증 코어는 정상 동작합니다.




기술 스택


Python — OOP 설계 (dataclass, ABC/abstractmethod, Strategy 패턴, 의존성 주입)
requests — 국세청 API 호출
openpyxl — 엑셀 리포트 출력
python-dotenv — API 키 환경변수 분리



설계 원칙


결정론적 코어 우선: 외부 의존성(API) 없이 합성 데이터만으로 모든 판정이 동작하도록 코어를 먼저 완성하고, 거래처 검증은 그 위에 어댑터로 얹는 구조
확장 가능한 검증 구조: 검사·탐지 클래스가 서로 독립적이라 새 신호 추가가 쉬움
외부 의존성 격리: 거래처 검증을 추상 인터페이스 + 다중 구현으로 설계해, 단일 외부 의존성에 묶이지 않고 폴백으로 견고성 확보
